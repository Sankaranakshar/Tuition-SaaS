## 2b. Stage 3 item 1 — Schedule workspace rebuild — DONE (2026-07-11)

One workspace (REDESIGN §6.1) replacing Calendar.tsx (967 lines, month-grid, native HTML5 drag), Bookings.tsx (mock data, deleted outright), and Timetable.tsx (read-only student view). Mounted at `/app/schedule` (staff) and `/app/my-schedule` (student/parent self-view, same component, role-branched) with `calendar`/`timetable`/`bookings` kept as redirects. `src/lib/schedule.ts` (19 unit tests: grid time-math, overlap layout, client-side conflict pre-check, availability-outside-hours check, template payload builder) + `src/hooks/useSchedule.ts` (`useScheduleSessions`, `useMyScheduleSessions`, `useClassTemplates`, `useTutorAvailability`, all on the shared `useRealtimeList`) + `src/pages/Schedule.tsx` (pointer-event week grid — drag-create, drag-move, drag-resize; month view; availability dimming; recurring-edit scope prompt; one-popover class wizard; "find a gap" slot picker).

**The gap this rebuild specifically closed:** the old Calendar.tsx rescheduled sessions via a direct client `.update()` on `class_sessions`, which `class_sessions_update`'s RLS policy permits for any staff member with zero conflict awareness — two sessions could silently overlap. New `PATCH /api/v1/scheduling/sessions/:id` and `PATCH /api/v1/scheduling/templates/:id` (server/routes/scheduling.ts) run the same advisory-lock + range-overlap check `checkTutorConflictAndInsert` already used at creation (refactored to accept an `excludeSessionId`), so reschedule is now genuinely server-authoritative. New `GET /api/v1/scheduling/gaps` powers "find a gap" against real `tutor_availability` + `class_sessions` data instead of the old client-side hardcoded-9-to-5 loop.

**Four real bugs found and fixed live this session, all via the actual drag/reschedule/scope-edit walkthrough (not caught by typecheck/unit/RLS — this class of bug never shows up until you actually drag something in a browser):**
1. **Drag-move used the day column captured at drag-*start*, never updated to the live pointer position** — dragging a session to a different day only changed its time-of-day and silently left it on the original day. Confirmed via a failed conflict test: server correctly found no overlap because the session had never actually moved days. Fixed in `handlePointerMove`'s "move" branch to use the pointer's live column every event, plus a rendering fix so the drag preview itself follows the pointer across day columns (it previously stayed pinned to the origin day's column throughout the drag).
2. **Out-of-order Realtime refetch race**: paging weeks or committing a reschedule can fire more than one `refetch()` in quick succession (the base mount effect plus an explicit week-change effect); network responses aren't guaranteed to resolve in request order, so a slower, stale request (e.g. the previous week) could resolve after a newer one and silently overwrite fresh data with stale results — manifested as the whole week grid going blank right after a successful action. Fixed with a monotonic sequence-number guard in the shared `useRealtimeList` hook (`src/hooks/useRealtimeList.ts`) so only the latest in-flight `refetch()` can commit its result — benefits every other workspace hook built on it, not just Schedule.
3. **`useRealtimeList`'s own mount effect only reruns on `[orgId, table]`**, never on the `load` closure changing — so paging `weekStart`/`weekEnd` never re-queried at all until this was given an explicit `refetch()` on week change in `useScheduleSessions`/`useMyScheduleSessions`.
4. **`PATCH /scheduling/templates/:id`'s "this and future sessions" scope-edit 500'd** (`column "updated_at" of relation "class_templates" does not exist`) — unlike `class_sessions`, which has an `updated_at` column live despite no migration file ever adding it (pre-existing schema drift, not this session's doing, worth a follow-up audit), `class_templates` genuinely never got one. Removed the stray `updated_at = now()` from the UPDATE statement.

## 7. Architecture Improvements

- **Module boundaries are healthy; keep them.** Pure logic in `src/lib/today.ts` and `server/utils/*` (unit-tested, IO-free) is the pattern to extend: every new workspace gets a pure core module + a thin page.
- **Shared types:** ~~there is no shared type package between `server/` and `src/`~~ **DONE (2026-07-10, HANDOFF §18.2; extended 2026-07-11).** `shared/schemas/billing.ts`, `shared/schemas/scheduling.ts`, and `shared/schemas/inbox.ts` cover those three domains' request/response shapes; server routes validate with them, client infers types from them. Still on inline Zod (audit note, 2026-07-11): `students.ts`, `parents.ts`, `members.ts`, `gateway.ts`, `documents.ts`, `settings.ts` — migrate each to `shared/schemas/` the next time its contract is touched, not as a big-bang pass.
- **API organization:** routes are cleanly split by domain. When Epic 7 lands, put the channel router in `server/jobs/` rather than routes, and formalize the cron surface (`/api/cron/*`) with a job registry.
- **Data access on the client:** pages talk to `supabase-js` directly with ad-hoc queries. Extract per-entity query hooks (`useSessions`, `useInvoices`) so Realtime subscription, bounding, and error handling live in one place each; adopt during Stage 2 rebuilds, not as a big-bang refactor.
- **Testing strategy:** keep the three-layer pyramid (unit for money math and today-derivations, PGlite RLS suite for authorization, supertest for route contracts). Add Playwright E2E for the five golden journeys once a seeded staging exists; E2E is the only layer that can catch the "never ran in a browser" class of bug.
- **CI/CD:** CI is solid (typecheck, unit, RLS, build, no external deps). Add: bundle-size check, `npm audit` gate, deploy-to-staging on merge once hosting exists, migration-ordering lint (applying 0001..N to a fresh PGlite already happens in the RLS suite, which is an excellent migration test; keep it mandatory).
- **Deployment:** now live (Vercel + Supabase Cloud, HANDOFF §13). Still needed: a written smoke script (`/api/health`, login, one read per table group) to run after every deploy, and a documented rollback procedure (Vercel's instant-rollback-to-previous-deployment covers the app side; there's no equivalent rehearsed procedure for a bad migration on the Supabase side yet).
